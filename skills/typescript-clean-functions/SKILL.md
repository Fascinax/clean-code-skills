# Clean Functions (TypeScript)

## F1: Too Many Arguments (Maximum 2-3)

```typescript
// Bad - too many parameters
function createUser(
  name: string,
  email: string,
  age: number,
  country: string,
  timezone: string,
  language: string,
  newsletter: boolean
): User { ... }

// Good - use object destructuring
type CreateUserParams = {
  name: string;
  email: string;
  age: number;
  country: string;
  timezone: string;
  language: string;
  newsletter: boolean;
};

function createUser({ name, email, age, ...rest }: CreateUserParams): User {
  ...
}
```

Object destructuring also makes call sites self-documenting:

```typescript
// Clear what each argument means
createUser({ name: 'Alice', email: 'alice@example.com', age: 30, ... });
```

## F2: No Output Arguments

Don't modify arguments as side effects. Return values instead.

```typescript
// Bad - modifies argument
function appendFooter(report: string[]): void {
  report.push('---\nGenerated by System');
}

// Good - returns new value
function withFooter(report: ReadonlyArray<string>): string[] {
  return [...report, '---\nGenerated by System'];
}

// Bad - mutates input array
function sortUsers(users: User[]): void {
  users.sort((a, b) => a.name.localeCompare(b.name));
}

// Good - returns new sorted array
function sortedByName(users: ReadonlyArray<User>): User[] {
  return [...users].sort((a, b) => a.name.localeCompare(b.name));
}
```

## F3: No Flag Arguments

Boolean flags mean your function does at least two things.

```typescript
// Bad - function does two different things
function render(isTest: boolean): void {
  if (isTest) {
    renderTestPage();
  } else {
    renderProductionPage();
  }
}

// Good - split into two functions
function renderTestPage(): void { ... }
function renderProductionPage(): void { ... }

// Bad - flag changes behavior
function createReport(detailed: boolean): Report { ... }

// Good - separate functions
function createSummaryReport(): Report { ... }
function createDetailedReport(): Report { ... }
```

## F4: Delete Dead Functions

If it's not called and not exported, delete it. No "just in case" code.
Git preserves history.

## Modern TypeScript Function Idioms

```typescript
// Use overloads for type-safe polymorphism
function parse(input: string): number;
function parse(input: string[]): number[];
function parse(input: string | string[]): number | number[] {
  if (Array.isArray(input)) return input.map(Number);
  return Number(input);
}

// Use generics with constraints for reusable functions
function getProperty<TObj, TKey extends keyof TObj>(
  obj: TObj,
  key: TKey
): TObj[TKey] {
  return obj[key];
}

// Use discriminated unions instead of overloads when possible
type FetchOptions =
  | { method: 'GET'; url: string }
  | { method: 'POST'; url: string; body: unknown };

async function fetchData(options: FetchOptions): Promise<Response> {
  switch (options.method) {
    case 'GET': return fetch(options.url);
    case 'POST': return fetch(options.url, { method: 'POST', body: JSON.stringify(options.body) });
  }
}

// Prefer functional patterns
// Bad - imperative loop
function getActiveEmails(users: ReadonlyArray<User>): string[] {
  const result: string[] = [];
  for (const user of users) {
    if (user.isActive) {
      result.push(user.email);
    }
  }
  return result;
}

// Good - declarative
function getActiveEmails(users: ReadonlyArray<User>): string[] {
  return users
    .filter(user => user.isActive)
    .map(user => user.email);
}

// Use async/await over raw Promises
// Bad
function getUser(id: string): Promise<User> {
  return fetch(`/api/users/${id}`)
    .then(res => res.json())
    .then(data => data as User);
}

// Good
async function getUser(id: string): Promise<User> {
  const response = await fetch(`/api/users/${id}`);
  return await response.json() as User;
}
```

## Quick Reference

| Rule | Principle | Key Signal |
|------|-----------|------------|
| F1 | Max 2-3 arguments | Use object destructuring for more |
| F2 | No output arguments | Return values, use `ReadonlyArray` |
| F3 | No flag arguments | Split into separate functions |
| F4 | Delete dead functions | If uncalled and unexported, remove it |

## AI Behavior

When reviewing functions, cite the rule number (e.g., "F1 violation: 5 positional arguments, use object destructuring").
When refactoring, explain the improvement (e.g., "Replaced 5 params with `CreateUserParams` type (F1)").
When applying Modern TypeScript idioms, infer the TS version from project config first (4.5+: inline `import type`; 4.9+: `satisfies`; 5.0+: `const` type params; default: 4.5).
