# Clean Functions (JavaScript)

## F1: Too Many Arguments (Maximum 2-3)

```javascript
// Bad - too many parameters
function createMenu(title, body, buttonText, cancellable) { ... }

// Good - use object destructuring
function createMenu({ title, body, buttonText, cancellable = false }) {
  ...
}

// Call site is now self-documenting
createMenu({
  title: 'Settings',
  body: 'Configure your preferences',
  buttonText: 'Save',
  cancellable: true,
});
```

More than 2-3 arguments means your function is doing too much or needs a data structure.

## F2: No Output Arguments

Don't modify arguments as side effects. Return values instead.

```javascript
// Bad - modifies argument
function appendFooter(report) {
  report.push('---\nGenerated by System');
}

// Good - returns new value
function withFooter(report) {
  return [...report, '---\nGenerated by System'];
}

// Bad - mutates the input array
function sortUsers(users) {
  users.sort((a, b) => a.name.localeCompare(b.name));
}

// Good - returns a new sorted array
function sortedByName(users) {
  return [...users].sort((a, b) => a.name.localeCompare(b.name));
}
```

## F3: No Flag Arguments

Boolean flags mean your function does at least two things.

```javascript
// Bad - function does two different things
function render(isTest) {
  if (isTest) {
    renderTestPage();
  } else {
    renderProductionPage();
  }
}

// Good - split into two functions
function renderTestPage() { ... }
function renderProductionPage() { ... }
```

## F4: Delete Dead Functions

If it's not called and not exported, delete it. No "just in case" code.
Git preserves history.

## Modern JavaScript Function Idioms

```javascript
// Prefer async/await over Promise chains (JS3)
// Bad
function getUser(id) {
  return fetch(`/api/users/${id}`)
    .then(res => res.json())
    .then(user => user)
    .catch(err => { throw err; });
}

// Good
async function getUser(id) {
  const response = await fetch(`/api/users/${id}`);
  return await response.json();
}

// Prefer functional patterns over imperative loops (JS5)
// Bad
function getActiveEmails(users) {
  const emails = [];
  for (let i = 0; i < users.length; i++) {
    if (users[i].isActive) {
      emails.push(users[i].email);
    }
  }
  return emails;
}

// Good
function getActiveEmails(users) {
  return users
    .filter(user => user.isActive)
    .map(user => user.email);
}

// Use default parameters instead of short-circuiting (JS7)
// Bad
function createUser(name, role) {
  const userRole = role || 'member';  // Bug: '' or 0 would be overridden
  ...
}

// Good
function createUser(name, role = 'member') { ... }

// Use method chaining for fluent APIs
class QueryBuilder {
  #filters = [];
  #sorts = [];

  where(condition) {
    this.#filters.push(condition);
    return this;
  }

  orderBy(field) {
    this.#sorts.push(field);
    return this;
  }

  build() {
    return { filters: this.#filters, sorts: this.#sorts };
  }
}

// Usage
const query = new QueryBuilder()
  .where({ status: 'active' })
  .orderBy('name')
  .build();
```

## Quick Reference

| Rule | Principle | Key Signal |
|------|-----------|------------|
| F1 | Max 2-3 arguments | Use object destructuring for more |
| F2 | No output arguments | Return values, use spread `[...]` |
| F3 | No flag arguments | Split into separate functions |
| F4 | Delete dead functions | If uncalled and unexported, remove it |

## AI Behavior

When reviewing functions, cite the rule number (e.g., "F1 violation: 4 positional arguments, use object destructuring").
When refactoring, explain the improvement (e.g., "Replaced 4 params with options object (F1, JS2)").
When applying Modern JavaScript idioms, infer the Node/ES version from project config first (ES2020+: `?.`, `??`; ES2022+: class fields `#`, top-level `await`; default: ES2020).
