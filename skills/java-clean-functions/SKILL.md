# Clean Functions (Java)

## F1: Too Many Arguments (Maximum 3)

```java
// Bad - too many parameters
public User createUser(String name, String email, int age,
                       String country, String timezone,
                       String language, boolean newsletter) {
    ...
}

// Good - use a parameter object or record
public record CreateUserRequest(
    String name,
    String email,
    int age,
    String country,
    String timezone,
    String language,
    boolean newsletter
) {}

public User createUser(CreateUserRequest request) {
    ...
}

// Good - use Builder pattern for complex construction
User user = User.builder()
    .name("Alice")
    .email("alice@example.com")
    .age(30)
    .build();
```

More than 3 arguments means your method is doing too much or needs a data structure.

## F2: No Output Arguments

Don't modify arguments as side effects. Return values instead.

```java
// Bad - modifies argument
public void appendFooter(StringBuilder report) {
    report.append("\n---\nGenerated by System");
}

// Good - returns new value
public String withFooter(String report) {
    return report + "\n---\nGenerated by System";
}

// Bad - mutates the input list
public void sortUsers(List<User> users) {
    users.sort(Comparator.comparing(User::name));
}

// Good - returns a new sorted list
public List<User> sortedByName(List<User> users) {
    return users.stream()
        .sorted(Comparator.comparing(User::name))
        .toList();
}
```

## F3: No Flag Arguments

Boolean flags mean your method does at least two things.

```java
// Bad - method does two different things
public void render(boolean isTest) {
    if (isTest) {
        renderTestPage();
    } else {
        renderProductionPage();
    }
}

// Good - split into two methods
public void renderTestPage() { ... }
public void renderProductionPage() { ... }

// Bad - flag argument in constructor
public Report(boolean detailed) { ... }

// Good - static factory methods
public static Report summary() { ... }
public static Report detailed() { ... }
```

## F4: Delete Dead Methods

If it's not called, delete it. No "just in case" code. Git preserves history.

## Modern Java Function Idioms

```java
// Use Optional chaining instead of null checks
public String getUserDisplayName(Long userId) {
    return userRepository.findById(userId)
        .map(User::displayName)
        .orElse("Unknown User");
}

// Use method references for concise lambdas
List<String> names = users.stream()
    .map(User::name)
    .filter(Predicate.not(String::isBlank))
    .toList();

// Use switch expressions instead of if/else chains
public String formatStatus(OrderStatus status) {
    return switch (status) {
        case PENDING -> "Awaiting confirmation";
        case SHIPPED -> "On its way";
        case DELIVERED -> "Received";
        case CANCELLED -> "Cancelled";
    };
}

// Use records for local data grouping (method-level)
public SummaryReport generateReport(List<Order> orders) {
    record OrderStats(long count, BigDecimal total) {}

    var stats = new OrderStats(
        orders.size(),
        orders.stream()
            .map(Order::total)
            .reduce(BigDecimal.ZERO, BigDecimal::add)
    );

    return buildReport(stats.count(), stats.total());
}
```

## Quick Reference

| Rule | Principle | Key Signal |
|------|-----------|------------|
| F1 | Max 3 arguments | Use `record` or Builder for more |
| F2 | No output arguments | Return values, don't mutate inputs |
| F3 | No flag arguments | Split into separate methods or use static factory |
| F4 | Delete dead methods | If uncalled, remove it |

## AI Behavior

When reviewing methods, cite the rule number (e.g., "F3 violation: boolean flag splits behavior").
When refactoring, explain the improvement (e.g., "Replaced constructor flag with static factory methods `Report.summary()` and `Report.detailed()` (F3)").
